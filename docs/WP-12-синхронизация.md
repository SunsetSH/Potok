# WP-12: Синхронизация между устройствами

Статус: план (proposed), к реализации не начат.

Основано на sync-ready фундаменте из ADR-008 и приватность-контракте ADR-007
(оба — в истории git, каталог `docs/` был очищен; решения продолжают
действовать). Фоновая загрузка — решение, помеченное «ADR-013» в комментариях
кода (`model_file_downloader.dart`, `main.dart`).

Цель: пользователь в настройках выбирает **один** из двух бэкендов
синхронизации — **Google Диск** или **свой сервер** — либо оставляет
синхронизацию выключенной (по умолчанию). Оба варианта делят один и тот же
клиентский движок; отличается только транспорт.

---

## 1. Что уже готово (не переделывать)

Схема БД спроектирована под синхронизацию с первого дня:

- **UUIDv7** у всех сущностей — идентификаторы не конфликтуют между
  устройствами без координации.
- **`OperationJournal`** — append-only журнал мутаций заметок/проектов/тегов:
  `operation_id`, `device_id`, `entity_kind`, `entity_id`, `base_revision`,
  `new_revision`, `operation_kind`, `occurred_at_utc`, минимальный payload.
  Идемпотентность повторов — по `operation_id`.
- **`revision`** у каждой сущности — optimistic concurrency, уже используется
  в каждой мутации (`notes_service`, `tags_service`).
- **Tombstone** через `deleted_at_utc` — удалённая запись физически не
  выпиливается до очистки корзины, поэтому удаление реплицируется как факт.
- **`device_id`** — генерируется при первом запуске, хранится локально.
- **`NoteEvent`** history — append-only.

Из этого следует: **~80% работы (журнал операций, конфликты, реплей) не
зависит от выбранного бэкенда**. Бэкенд — это только «куда положить
неотправленные записи журнала + медиа и откуда забрать чужие».

## 2. Общая архитектура (одинакова для обоих бэкендов)

### 2.1. Модель синхронизации: журнал операций + CAS-хранилище блобов

Синхронизируются **две вещи**:

1. **Записи `OperationJournal`** — маленькие JSON-документы. Это источник
   истины «что произошло». Реплей записи на другом устройстве применяет ту же
   мутацию через существующий revision-guard.
2. **Медиа-блобы** (аудио, изображения) — крупные бинарные файлы, адресуются по
   **SHA-256** (content-addressed, тот же паттерн, что уже используется для
   ASR-моделей и финализации аудио). Блоб иммутабелен: один хеш — одно
   содержимое, дедупликация и целостность бесплатно.

### 2.2. Абстракция бэкенда

```dart
abstract interface class SyncBackend {
  /// Загрузить новые записи журнала этого устройства.
  Future<void> pushOperations(List<OperationRecord> ops);

  /// Забрать записи журнала других устройств после курсора.
  Future<SyncPullResult> pullOperations(SyncCursor after);

  /// Загрузить блоб (идемпотентно по sha256 — уже есть → no-op).
  Future<void> pushBlob(String sha256, Stream<List<int>> bytes);

  /// Скачать блоб по хешу (по запросу, не автоматически на все устройства).
  Future<Stream<List<int>>> pullBlob(String sha256);

  Future<bool> isAvailable();
}
```

`GoogleDriveSyncBackend` и `SelfHostedSyncBackend` реализуют этот интерфейс.
Выбор в настройках хранится в `app_meta` (`sync.backend` = `off|gdrive|server`).
Верхнеуровневый `SyncService` не знает, какой бэкенд активен.

### 2.3. Цикл синхронизации `SyncService`

1. **Push**: выбрать записи `OperationJournal`, ещё не подтверждённые
   бэкендом (курсор отправки на устройство), загрузить их + приложить блобы,
   на которые они ссылаются и которых ещё нет в хранилище.
2. **Pull**: забрать чужие записи после курсора приёма, отсортировать по
   `(occurred_at_utc, operation_id)`, **реплеить по одной** через тот же
   revision-guard, что и локальные мутации.
3. Идемпотентность: запись с уже применённым `operation_id` пропускается
   (у каждого устройства — таблица применённых `operation_id`).

Синхронизация **не realtime** и не обязана быть мгновенной: периодический
триггер (при старте, при уходе в фон, по кнопке «синхронизировать сейчас»,
опционально по таймеру). На сервере (вариант B) можно добавить push-канал.

### 2.4. Конфликты

`base_revision` записи расходится с текущей локальной ревизией сущности →
конфликт. Политика — **не тихий last-write-wins**:

- Для заметок: сохранить **конфликтную копию** (как Dropbox/Syncthing) —
  вторую заметку с пометкой «конфликт, устройство X, время T». Пользователь
  разрешает вручную. Ничего не теряется молча.
- Для тегов/проектов (переименование, цвет): last-write-wins по
  `occurred_at_utc` приемлемо (низкая ценность, редкие конфликты), но факт
  логируется.

### 2.5. Медиа: выборочно, не «всё везде»

Блобы **не** тянутся автоматически на каждое устройство: телефон не должен
внезапно заполниться часами аудио с ПК. Записи журнала (с метаданными и
хешами) реплицируются всегда и дёшево; сам медиафайл скачивается **по
запросу** (пользователь открыл заметку / нажал «загрузить аудио»), с
индикатором. Хеш из журнала гарантирует целостность после скачивания.

### 2.6. Фоновая синхронизация на Android

Та же проблема, что решалась для загрузки моделей (ADR-013): таймер в
Dart-изоляте останавливается, когда ОС приостанавливает приложение. Крупные
push/pull блобов должны идти через нативный фоновый механизм
(`background_downloader`/WorkManager), а не обычный `HttpClient` в UI-изоляте.
Мелкие записи журнала можно синхронизировать на переднем плане.

## 3. Вариант A — Google Диск

### 3.1. Как работает

- Аутентификация через **Google Sign-In + OAuth**, scope
  `drive.file` (доступ только к файлам, созданным приложением — не ко всему
  диску пользователя).
- Приложение создаёт папку `Поток/` в Диске пользователя (или использует
  **appDataFolder** — скрытую папку приложения, невидимую пользователю в UI
  Диска, что чище).
- Записи журнала — маленькие объекты (один файл на устройство-пачку или
  append-подобная структура). Блобы — файлы с именем = sha256.
- Устройство периодически загружает свои новые записи и скачивает чужие
  (Drive Changes API / список файлов по времени модификации).

### 3.2. Что нужно реализовать

1. OAuth-флоу (`google_sign_in`) + консоль Google Cloud (OAuth client,
   верификация приложения для scope `drive.file`, экран согласия).
2. `GoogleDriveSyncBackend` поверх Drive REST API (`googleapis` пакет).
3. Разбиение журнала на объекты + курсоры чтения/записи.
4. Обработка квот/ошибок Drive (rate limit, 403, отзыв доступа).

### 3.3. Плюсы

- Не нужно администрировать сервер; надёжность и бэкап — на стороне Google.
- У большинства пользователей уже есть Google-аккаунт.
- Работает «из коробки» на любых сетях, без VPN/проброса портов.

### 3.4. Минусы и риски

- **Противоречит текущему приватность-контракту (ADR-007: офлайн-онли).**
  Содержимое заметок оказывается на стороннем сервере; без клиентского
  шифрования Google технически может его прочитать. → Нужно либо явно
  расширить ADR-007, либо добавить сквозное шифрование блобов/записей
  (существенный доп. объём).
- Верификация приложения Google для OAuth (процесс ревью, может занять
  недели; для `drive.file`/`appDataFolder` — проще, но всё равно есть).
- Не realtime: Drive не даёт дешёвого push для фонового мобильного клиента.
- Зависимость от политики/квот/доступности Google, риск блокировок по
  региону.

## 4. Вариант B — свой сервер (Linux-бокс пользователя)

### 4.1. Как работает

- Небольшой сервис на **Dart `shelf`** (один язык с клиентом — переиспользуются
  доменные модели) с двумя группами эндпоинтов:
  - `POST /operations` (push записей журнала), `GET /operations?after=cursor`
    (pull чужих записей);
  - `PUT /blob/{sha256}` (идемпотентно), `GET /blob/{sha256}`.
- Сервер — «глупое» хранилище: не разбирает схему заметок, только хранит
  записи журнала и блобы по хешу. Вся бизнес-логика и разрешение конфликтов —
  на клиентах.
- Транспорт: **не пробрасывать порт наружу**. Поднять **Tailscale/WireGuard**
  между устройствами и сервером — сервер виден как узел приватной сети, без
  публичного порта, без возни с TLS-сертификатами, минимальная поверхность
  атаки. (Альтернатива — публичный HTTPS + reverse-proxy + Let's Encrypt, но
  это больше обслуживания и риска.)
- Аутентификация: **токен на устройство** (не один общий секрет — потерянный
  телефон отзывается индивидуально, без перевыпуска токенов всем).
- Опционально: push-канал (WebSocket/SSE) для почти мгновенной синхронизации,
  раз ограничений квот нет.

### 4.2. Что нужно реализовать

1. **Сервер** (`shelf`): эндпоинты operations/blob, хранилище (журнал —
   SQLite/файлы, блобы — файлы по хешу), пер-устройственные токены, простой
   healthcheck.
2. **Клиент**: `SelfHostedSyncBackend` (HTTP + токен), настройка адреса
   сервера и токена в UI.
3. **Изоляция на сервере**: отдельный Docker-контейнер, непривилегированный
   пользователь, resource limits — чтобы баг в sync-сервисе не стал вектором к
   умному дому на той же машине.
4. Бэкап данных сервера (журнал + блобы) — ответственность владельца, описать
   в README сервера.

### 4.3. Плюсы

- **Полный контроль над данными**, соответствует духу проекта
  (офлайн/приватность-first) — данные не покидают инфраструктуру пользователя.
- Можно сделать почти мгновенную синхронизацию (свой push, без квот).
- Инфраструктура уже есть (Linux-бокс с умным домом).
- Нет процесса ревью/верификации у третьей стороны.

### 4.4. Минусы и риски

- Доступность синхронизации зависит от домашнего сервера (не 24/7 SLA; свет,
  сеть, перезагрузки).
- Сосед по машине — умный дом: обязательна изоляция (см. 4.2.3), иначе
  расширяется поверхность атаки на критичную систему.
- Всё обслуживание (патчи ОС, бэкапы, мониторинг, обновление сервиса) — на
  владельце.
- Мобильный клиент вне домашней сети требует VPN (Tailscale) — небольшой порог
  настройки для пользователя.

## 5. Общие клиентские задачи (независимо от бэкенда)

1. `SyncService` — читает неотправленные `OperationJournal`, пушит + прикладывает
   блобы; тянет чужие операции и реплеит через revision-guard.
2. Таблицы курсоров (отправки/приёма) и применённых `operation_id`.
3. Разрешение конфликтов: конфликтные копии заметок (4 в разделе 2.4).
4. Выборочная загрузка медиа по запросу + индикатор (2.5).
5. Фоновая синхронизация блобов через нативный механизм (2.6).
6. UI настроек: выбор бэкенда (off/gdrive/server), статус («последняя
   синхронизация…», «N ожидает отправки»), кнопка «синхронизировать сейчас»,
   разрешение конфликтов.
7. Метрики устойчивости: повтор по `operation_id` идемпотентен; частичный сбой
   (упал на середине блоба) возобновляется, не дублирует.

## 6. Приватность

- Вариант B по умолчанию **не нарушает** ADR-007 (данные не уходят третьей
  стороне), но добавляет сеть — контракт нужно уточнить: «синхронизация —
  второе явное, opt-in сетевое использование, только на адрес, заданный
  пользователем».
- Вариант A **требует** пересмотра ADR-007 (данные у Google) и/или сквозного
  шифрования. Это решение продукта — зафиксировать отдельным ADR **до**
  реализации варианта A.

## 7. Этапы

1. **ADR-014**: зафиксировать модель синхронизации (журнал+CAS, конфликтные
   копии, приватность обоих вариантов) — до кода.
2. **Клиентское ядро**: `SyncService` + `SyncBackend` + курсоры + реплей +
   конфликтные копии. Тесты на реплей/идемпотентность/конфликты с фейковым
   in-memory бэкендом.
3. **Вариант B** (рекомендуется первым — проще, приватнее, вся инфраструктура
   есть): сервер на `shelf` + `SelfHostedSyncBackend` + Tailscale-инструкция +
   Docker-изоляция.
4. **Вариант A** (при спросе): OAuth + `GoogleDriveSyncBackend` + пересмотр
   ADR-007/шифрование.
5. Фоновая синхронизация блобов на Android.

## 8. Рекомендация

Начать с **варианта B (свой сервер)**: он лучше соответствует заявленной
приватность-политике проекта, укладывается в уже имеющуюся инфраструктуру
пользователя и не требует внешнего ревью. Абстракция `SyncBackend` оставляет
дверь для варианта A открытой без переделки ядра.

Это крупная фича, сравнимая по объёму с моделями/загрузкой; ядро (раздел 5)
переиспользуется обоими бэкендами и должно быть реализовано и покрыто тестами
до подключения любого транспорта.

## 9. Открытые вопросы

- Сквозное шифрование: делать ли его сразу для обоих бэкендов (тогда и Google
  безопаснее), или только при выборе варианта A?
- Гранулярность записей журнала на Google Диске (один файл на операцию — много
  мелких файлов и API-вызовов; батч — сложнее курсоры).
- Формат конфликтной копии в UI (отдельная заметка vs встроенный diff).
- Нужен ли таймерный автосинк или достаточно триггеров (старт/фон/кнопка)?
