# ADR-011 — кроссплатформенный буфер изображений

Статус: принято  
Дата: 2026-07-18

## Контекст

`FR-DOC-003` требует inline images как managed media. Flutter `Clipboard`
предоставляет переносимый API только для текста, а Potok должен вставлять
изображение по явному Ctrl+V на Windows и Android. Прямой Win32 implementation
оставил бы Android без эквивалента; отдельные method channels дублировали бы
декодирование platform clipboard formats.

## Решение

Использовать Apache-2.0 пакет `pasteboard 0.5.0` через узкий
`ClipboardImageReader` adapter. Он поддерживает Android/Windows и возвращает
байты platform-представления. Адаптер не доверяет заявленному
расширению: он определяет PNG/JPEG/WebP/BMP по magic bytes. Windows
clipboard DIB фактически читается пакетом как BMP, а Android может
вернуть исходный JPEG/PNG. Копирование файла из Windows Explorer
обрабатывается через file-list fallback с проверкой размера до чтения.

Источник пакета и platform matrix:
<https://pub.dev/packages/pasteboard>.
Application/UI видят только bounded bytes и формат; platform API не попадает в
Domain. Чтение происходит только вследствие явного paste action.

Полученные bytes не встраиваются в document и не пишутся во временный каталог
UI. `ImagesService` проверяет лимит и magic bytes и публикует их тем же
staging/atomic media protocol, что и picker. Текстовый paste остаётся отдельной
веткой и не передаётся image pipeline.

Potok использует только read API. Android `FileProvider`, требуемый пакету для
записи файлов в clipboard, не добавляется: экспортировать внешние пути и писать
clipboard этот WP не должен. Release size измеряется после интеграции.

## Последствия

- единый adapter и одинаковые проверки Windows/Android;
- Windows screenshots/DIB доступны как валидный BMP без собственного GDI decoder;
- версия зависимости закреплена, включается в SBOM и повторно проверяется при
  обновлении;
- clipboard bytes кратковременно находятся в памяти в незашифрованном виде, как
  и выбранный пользователем файл; содержимое и пути не логируются;
- если platform clipboard недоступен, обычная работа редактора и picker остаются
  доступны.

Первоначально рассматривался `super_clipboard 0.9.1`, но Android release на
текущем Gradle 9 не собирается: его CargoKit использует удалённый `Project.exec`.
Поэтому этот вариант отклонён после build spike, а не оставлен как скрытый
release blocker.

## Отклонённые варианты

- только `Clipboard.getData`: не читает бинарные изображения;
- Windows-only Win32 plugin: нарушает обязательный Android parity;
- собственные native channels: больше недоверенного platform-кода и тестовой
  поверхности без продуктовой выгоды;
- base64 в document: нарушает media/domain инварианты.
