# ADR-016: гарантированная упаковка SQLite в Android release

Статус: принято 2026-07-23.

## Контекст

`drift_flutter 0.3.1` использует `sqlite3 3.x`, который поставляет SQLite через
Dart Native Assets. В Flutter 3.44 после чередования `flutter test`, Windows- и
Android-сборок встречается состояние, в котором hook `sqlite3` успешно создаёт
ARM64 `libsqlite3.so` и `native_assets.json` ссылается на библиотеку, но Flutter
оставляет Android staging-каталог `jniLibs` пустым. Полученный APK собирается без
ошибки, однако приложение не может открыть Drift до первого кадра.

Предыдущее удаление сгенерированного `native_assets.json` перед release-задачей
не гарантировало повторную установку библиотеки и поэтому не устраняло дефект.

## Решение

- Release-задача Android после `compileFlutterBuildRelease` находит последний
  успешный hook-output `sqlite3` для каждой запрошенной Android ABI.
- Библиотеки проверяются на существование и минимальный ожидаемый размер, затем
  копируются в соответствующие Flutter release `jniLibs` до merge native
  libraries. Компактная сборка `android-arm64` по-прежнему содержит только
  `arm64-v8a`.
- `assembleRelease` открывает каждый полученный APK и для каждой ABI с
  `libapp.so` прерывает сборку, если рядом нет `libsqlite3.so`.
- Release build number увеличивается, чтобы исправленный APK устанавливался как
  обновление поверх проблемной версии.

Сгенерированная библиотека остаётся в `.dart_tool`/`build` и не добавляется в
Git. Источник и версия SQLite по-прежнему фиксируются `pubspec.lock`.

## Последствия

Сборка больше не может завершиться успешно с ARM64 APK, который заведомо упадёт
из-за отсутствующей SQLite. Решение локализовано в Android release pipeline и не
меняет доменную модель, формат базы или Windows-сборку. При изменении структуры
Dart Native Assets либо Flutter staging сборка остановится с явным сообщением,
а не выпустит повреждённый APK.
