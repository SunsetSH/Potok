# ADR-013: фоновая загрузка ASR-моделей

Статус: принято, уточнено 2026-07-22.

## Контекст

ASR-паки занимают сотни мегабайт. Обычный `HttpClient` в UI-изоляте не даёт
надёжной докачки на Android при сворачивании, блокировке экрана и остановке
Flutter-движка. При этом загруженные файлы нельзя публиковать как модель до
проверки манифеста, совместимости и SHA-256.

## Решение

- Android/iOS загружают файлы через нативные задачи `background_downloader`;
  desktop продолжает использовать `HttpClient`.
- На Android каждый файл ASR-пака выполняется как foreground `dataSync` с
  уведомлением, включая маленькие `tokens`/`joiner`: при screen lock система
  может ограничить сеть именно в промежутке между большими файлами. Временные
  файлы размещаются в `applicationSupport`, Android cache для model pack
  отключён из-за квоты и ненадёжного resume больших файлов.
- Android manifest сохраняет normal-permission `ACCESS_NETWORK_STATE`:
  WorkManager требует его при постановке jobs с connectivity constraint.
  Разрешение не добавляет новых сетевых путей и не меняет offline-инференс.
- У каждого файла стабильный task ID `<model_id>::<file_name>`. БД задач
  отслеживается между запусками; startup обязан выполнить единый
  `FileDownloader.start()`, включая импорт событий, накопленных в фоне.
- Намерение скачать pack записывается в `app_meta` до сетевого I/O и удаляется
  только после проверки, атомарной установки и активации модели. Поэтому
  завершённое Android-уведомление не может остаться «оторванным» от модели.
- Staging имеет стабильный путь `models/.downloads/<model_id>`, а не каталог в
  системном temp. При восстановлении каждый уже полученный файл сначала
  проверяется по SHA-256; валидный файл повторно не скачивается, повреждённый
  удаляется и ставится в очередь заново.
- Все файлы валидированного manifest передаются нативному загрузчику
  конкурентно. Последовательная постановка запрещена: если процесс погибал
  после первого файла, WorkManager раньше физически не знал об остальных.
- При старте приложения незавершённое намерение автоматически возобновляет
  загрузку/проверку/установку. Экран каталога показывает восстановление вместо
  возврата к ложному состоянию «Скачать».
- Полностью полученный staging устанавливается и активируется без повторного
  обращения к сети. Исходный URL хранится рядом со staging и должен совпасть с
  pending-намерением; это исключает публикацию чужого или устаревшего pack.
- Финальные неуспешные записи удаляются перед повторной постановкой задачи.
  После `enqueue` запрещено принимать старую финальную запись как результат
  новой попытки; завершение сверяется и по stream событий, и по БД.
- После асинхронного чтения БД повторно проверяется, не завершил ли тот же
  результат live callback. Это исключает гонку `Future already completed` на
  успешно скачанном файле.
- Нативный `updates` является single-subscription stream, поэтому приложение
  создаёт над ним один process-wide broadcast-мультиплексор. Все файлы пака и
  повторные попытки фильтруют события по стабильному task ID; прямые повторные
  подписки к нативному stream запрещены.
- Нативный файл после успешной загрузки копируется в staging установки.
  Активация происходит только после проверки allowlist исходного URL,
  совместимости manifest и SHA-256 всех файлов.

## Последствия и проверка

Сворачивание и screen lock не прерывают текущий нативный transfer. Если
Flutter не получил событие завершения в момент приостановки, оно восстанавливается
из native storage/БД после возобновления. Убийство процесса не публикует
частичный пак. Нативные задачи продолжают работать независимо, а установка
завершается после возвращения приложения и успешной проверки постоянного
staging. Просить исключение из оптимизации батареи или
`REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` не требуется: это не устраняет разрыв
оркестрации, ухудшает автономность и ограничивается правилами магазинов.

Device-smoke для релиза: начать загрузку каждой модели, заблокировать экран не
менее чем на две минуты, разблокировать и проверить продолжение прогресса;
повторить со сворачиванием и с перезапуском процесса во время загрузки.

Первичная проверка 2026-07-22 на OnePlus PLR110, Android 16 подтвердила, что
WorkManager продолжает transfer под lock screen. Поздний тест на другом
устройстве выявил разрыв: уведомление завершалось, но последовательная Dart-
оркестрация не публиковала pack после выгрузки процесса. Поэтому проверка с
перезапуском процесса является обязательной, а результат новой durable-
оркестрации нельзя считать device-verified до повторного smoke.
