## HIGH

### 1. Скрытые технические заметки утекают в «Корзину»
`app/lib/application/notes_service.dart:564-600, 703-752, 1643-1685`
Image-драфты (`beginImageNoteDraft`) и staged-аудио (`beginAudioNote`) скрываются через `deletedAtUtc`, а `watchTrash()`/`fetchTrashPage()`/`trash_count` выбирают все строки с `deletedAtUtc IS NOT NULL`. Во время записи аудио пользователь видит в корзине «заметку-призрак»; «восстановить» делает видимой заметку без готового ассета; счётчик корзины врёт.
**Фикс:** отдельный флаг/kind «служебная строка» вместо магии deletedAtUtc; фильтр во всех trash-запросах.

### 2. Перезапись бэкапа не атомарна — можно потерять единственный бэкап
`app/lib/application/backup_service.dart:152-154`
Существующая копия удаляется до `rename` нового zip. Сбой rename (антивирус/lock/диск) → старый бэкап уже уничтожен, `finally` удалит и новый temp. Комментарий класса обещает «atomic rename», его нет.
**Фикс:** rename старого в `.bak`, удалять только после успешного rename нового.

### 3. Экспорт падает на больших объёмах вместо усечения
`app/lib/application/export_service.dart:41-54, 162-165`
`collectNotes` может набрать до `maxNotes + 199` записей (проверка лимита до добавления страницы в 200), а `_loadContext` бросает `ArgumentError` при `selection.length > maxNotes`. При >10 000 заметок экспорт падает собственной же выборкой.
**Фикс:** усекать результат до `maxNotes` после цикла.

### 4. Необработанные ошибки в цепочке записи WAV
`app/lib/infrastructure/audio_recorder.dart:152`
`_writeQueue = _writeQueue.then((_) => file.writeFrom(chunk))` без обработчика. При сбое (диск переполнен): unhandled future error; цепочка «залипает», чанки теряются, `_wavDataBytes` продолжает расти → заголовок WAV объявляет больше данных, чем записано; `await _writeQueue` в `cancel()` бросает до удаления файла.
**Фикс:** try/catch вокруг write, флаг сбоя, остановка записи; в `stop()/cancel()` глотать ошибку очереди.

### 5. Потеря несохранённых правок при закрытии редактора (гонка dispose ↔ автосохранение)
`app/lib/presentation/note_detail_pane.dart:83-108, 196-275`
`dispose()` делает flush через `unawaited(updateDocument(...))`, не проверяя in-flight автосохранение (`_saving`/`_saveCompletion`). Конкурентный вызов со «старым» `note.revision` проигрывает гонку → `StateError` проглатывается в `catchError` → **правки молча теряются**. Плюс `_save()` при конкурентном вызове не перепроверяет `_dirty` после ожидания.
**Фикс:** в `dispose()` сериализовать финальный flush через `_saveCompletion` (как в `_flushNow`).

### 6. FTS5 привязан к нестабильному rowid
`app/lib/infrastructure/db/schema.drift:29-35`
`notes_fts` использует `content_rowid='rowid'`, а PK у `notes` — TEXT (rowid неявный). `VACUUM` может перенумеровать rowid → поиск молча возвращает чужие заметки.
**Фикс:** после любого VACUUM выполнять `INSERT INTO notes_fts(notes_fts) VALUES('rebuild')` (зафиксировать в коде обслуживания БД), либо суррогатный `INTEGER PRIMARY KEY`.

- `UuidV7Generator`, `Clock` — нет прямых юнит-тестов.
- `PotokDocument.decode`: невалидный JSON, отсутствующий `delta`, неверный `format`, `assetId` с недопустимыми символами — ветки ошибок не зафиксированы тестами.
- Глубокое копирование `ops` проверено лишь на 2 уровня вложенности.
- В `backup_restore_test.dart` drift предупреждает о повторном открытии `AppDatabase` на одном executor.

## Что сделано хорошо

- Optimistic-concurrency по `revision`, «одна мутация — одна транзакция» + журнал операций (sync-ready).
- Протокол `partial → hash → atomic rename` выдержан в MediaStore, AsrModelManager, BundledModelInstaller; path traversal закрыт в restore и манифестах; zip-bomb лимиты; sha256; экранирование FTS/LIKE; защита CSV от formula injection.
- Реентерабельная миграция v2→v3 (`DROP IF EXISTS` + rebuild FTS).
- В UI почти везде `mounted`-проверки после await, дебаунсы, версионирование (`revision`, `_noteToken`, `Completer`); продуманная сериализация плагин-вызовов в `windows_integration.dart`.
